# Frontend Audit -- Skeptic Review
Date: 2026-03-02

## Executive Summary

The Cassini frontend is a substantial codebase (~87k lines, ~195 files) that shows signs of rapid sprint-driven development. The core architecture (React Query for server state, Zustand for client state, fetchApi for auth) is sound, but 13 dead components totaling ~1,876 lines have accumulated as features were iterated on. The most concerning systemic issue is a pervasive pattern of leaking backend error messages directly into toast notifications (133 instances), which both violates the project's own CLAUDE.md security guidelines and exposes internal implementation details to users. The `quality.ts` hooks file has become a kitchen-sink dumping ground mixing anomaly, capability, MSA, FAI, and distribution concerns in a single 636-line file. Plant ID state is accessed through two competing mechanisms (`usePlantContext` vs `useUIStore.selectedPlantId`) with no enforcement of which to use where, creating a maintenance hazard.

### [ANTI-PATTERN] Backend Error Messages Leaked to Users via toast.error (133 instances)

**Severity**: HIGH
**Location**: All files in `api/hooks/*.ts` -- 133 occurrences
**Description**: Nearly every mutation hook follows this pattern:

```typescript
onError: (error: Error) => {
  toast.error(`Failed to save config: ${error.message}`)
}
```

The `error.message` comes from `fetchApi`, which extracts `error.detail` from the backend JSON response. This means raw backend messages (including potential SQL errors, stack traces, or internal route names) are displayed to end users. The project's own CLAUDE.md explicitly states: "NEVER pass `str(e)` to API clients -- log server-side, return generic messages."

**Evidence**: `grep -rn "toast.error" --include="*.tsx" --include="*.ts" . | grep "error.message" | wc -l` returns 133. Sample from `api/hooks/admin.ts`:
```typescript
onError: (error: Error) => {
  toast.error(`Failed to save config: ${error.message}`)  // line 51
}
```

**Knock-on Effects if Addressed**: Every mutation hook's `onError` handler would need to be updated to show a generic message while logging the actual error to console. This is ~133 changes across ~15 files, but the pattern is mechanical.

**Recommendation**: Create a shared `handleMutationError(context: string)` utility:
```typescript
function handleMutationError(context: string) {
  return (error: Error) => {
    console.error(`${context}:`, error.message)
    toast.error(`${context}. Please try again.`)
  }
}
```
Then replace all 133 instances. Consider using a global onError handler on the QueryClient for mutations.

### [ANTI-PATTERN] Plant ID Accessed Through Two Competing Mechanisms

**Severity**: HIGH
**Location**: 12 components via `useUIStore(s => s.selectedPlantId)`, 16 via `usePlantContext()`, 12 via `usePlant()`
**Description**: The selected plant can be accessed in three ways:
1. `usePlantContext()` / `usePlant()` -- the PlantProvider (React Context) which wraps the store
2. `useUIStore(s => s.selectedPlantId)` -- directly reading the Zustand store

These are functionally equivalent but create confusion about which is canonical. The PlantProvider adds validation (auto-selects first plant if selected one is stale) and provides the full Plant object with `isLoading`/`error` states. The raw store access skips all of this.

12 components bypass PlantProvider entirely, including all connectivity tab components (`MonitorTab`, `MappingTab`, `ServersTab`, `GagesTab`, `ProtocolSourceFields`, `ServerSelector`, `CharacteristicPicker`), the FAI page, the ERP integrations tab, and even `AuthProvider`.

**Evidence**: `grep -rn "useUIStore.*selectedPlantId" --include="*.tsx" . | grep -v "stores/uiStore\|providers/PlantProvider"` returns 12 distinct files.

**Knock-on Effects if Addressed**: All 12 components would need to be migrated to `usePlantContext()`. The key risk is that `usePlantContext` throws if used outside `PlantProvider` -- but all these components are already mounted within the provider tree (they're inside `RequireAuth > AuthenticatedProviders`).

**Recommendation**: Standardize on `usePlantContext()`. Add an ESLint rule or code review gate to prevent direct `useUIStore` access for `selectedPlantId`. The store should be treated as an internal detail of `PlantProvider`.

### [ANTI-PATTERN] `quality.ts` Hooks File is a Kitchen Sink (636 lines, 5 domains)

**Severity**: MEDIUM
**Location**: `api/hooks/quality.ts`
**Description**: This single file exports hooks for five unrelated domains:
1. Anomaly detection (config, events, status, dashboard, trigger)
2. Non-normal capability & distribution fitting
3. Capability & capability history
4. Rule presets
5. MSA studies & measurements (imports from `msa.api.ts`)
6. FAI reports & items (imports from `fai.api.ts`)

At 636 lines it is the largest hooks file, yet MSA and FAI have their own API files (`msa.api.ts`, `fai.api.ts`). The hooks for these should live in their own hook files too.

**Evidence**: `grep -n "useFAI\|useMSA\|fai\|msa" api/hooks/quality.ts | head -5` shows MSA/FAI hooks starting at line 291, importing from their respective API files but colocated with anomaly/capability hooks.

**Knock-on Effects if Addressed**: Would need to create `api/hooks/msa.ts` and `api/hooks/fai.ts`, move the relevant hooks, and update the barrel export. All consumer imports go through the barrel so they would not break.

**Recommendation**: Split into at least `api/hooks/anomaly.ts`, `api/hooks/capability.ts`, `api/hooks/msa.ts`, and `api/hooks/fai.ts`. The barrel re-export means no consumer changes are needed.

### [DESIGN] OperatorDashboard Has 14 Store Subscriptions and 12 useState Calls

**Severity**: MEDIUM
**Location**: `pages/OperatorDashboard.tsx` (702 lines)
**Description**: The OperatorDashboard subscribes to 14 individual `useDashboardStore` selectors and maintains 12 local `useState` instances. Each store selector creates a separate subscription, meaning the component potentially re-renders on any of 14 store changes. The `chartOptions` computation is an IIFE that runs on every render (not memoized).

**Evidence**:
```
grep -c "useDashboardStore" pages/OperatorDashboard.tsx  =>  14
grep -c "useState" pages/OperatorDashboard.tsx  =>  12
```

Lines 147-162 show the `chartOptions` IIFE:
```typescript
const chartOptions = (() => {
  // ... computes on every render
})()
```

**Knock-on Effects if Addressed**: Replacing the IIFE with `useMemo` keyed on `[timeRange, productCodeFilter]` would prevent unnecessary recalculation. Grouping related store selectors into a single selector (e.g., `const chartConfig = useDashboardStore(s => ({ timeRange: s.timeRange, showBrush: s.showBrush, ... }))`) would reduce subscription count but requires a shallow-equality compare.

**Recommendation**:
1. Wrap `chartOptions` in `useMemo`
2. Consider grouping related store selectors (chart-display vs modal-state vs toolbar-state)
3. Extract chart-area rendering into a sub-component to isolate re-renders

### [DESIGN] Many Connectivity Components Bypass Hooks Layer with Direct API Calls

**Severity**: LOW
**Location**: `components/connectivity/*.tsx`, `components/analytics/*.tsx`
**Description**: At least 18 components in the connectivity and analytics folders make direct `useQuery`/`useMutation` calls to API namespaces (`brokerApi`, `opcuaApi`, `characteristicApi`) instead of using the centralized hooks in `api/hooks/`. This creates inconsistency:
- No shared cache key management
- No shared error handling patterns
- No shared invalidation logic

**Evidence**: `grep -rn "characteristicApi\.\|brokerApi\.\|opcuaApi\." --include="*.tsx" . | grep -v "api/"` returns 18+ hits across connectivity components.

For example, `components/connectivity/BrokerStatusCards.tsx` (itself dead code) defines its own `useMutation` calls to `brokerApi.connect/disconnect` with inline query key strings, while the hooks layer in `api/hooks/connectivity.ts` already has centralized broker hooks.

**Knock-on Effects if Addressed**: Each component would need to be migrated to use centralized hooks, and some hooks may need to be created if they don't exist yet. Risk is low since the pattern is mechanical.

**Recommendation**: For new features, enforce hooks-layer usage. Incrementally migrate existing direct calls. Particularly important for query key consistency.

### [DESIGN] ErrorBoundary Missing from 10 of 14 Pages

**Severity**: MEDIUM
**Location**: All pages except `OperatorDashboard.tsx`, `WallDashboard.tsx`, `KioskView.tsx`
**Description**: Only 3 pages use `ErrorBoundary` (plus the top-level `RouteErrorBoundary` in `App.tsx`). The remaining 10 pages have no component-level error boundaries:

- DataEntryView, ViolationsView, MSAPage, FAIPage
- AnalyticsPage, DOEPage, ConfigurationView, ConnectivityPage
- SettingsView, GalaxyPage

The `RouteErrorBoundary` in `App.tsx` wraps the entire `<Layout />` tree, which catches errors but resets the entire page (not just the failing section). Page-level boundaries would allow graceful degradation.

**Evidence**: `grep -ql "ErrorBoundary" pages/*.tsx` returns only `OperatorDashboard.tsx`. The other pages have zero `ErrorBoundary` references.

**Knock-on Effects if Addressed**: Adding `ErrorBoundary` wrappers at the page level is low-risk. The existing `ErrorBoundary` component supports a `fallback` prop for custom recovery UI.

**Recommendation**: Wrap each page's main content in an `<ErrorBoundary>`. For pages with tabbed content (Analytics, Connectivity, Settings), wrap each tab independently so one failing tab doesn't take down the whole page.

### [QUALITY] Pervasive `any` Types in ECharts Integration (29 instances)

**Severity**: LOW
**Location**: `components/ControlChart.tsx`, `components/AttributeChart.tsx`, `components/CUSUMChart.tsx`, `components/EWMAChart.tsx`, `components/DistributionHistogram.tsx`, `components/analytics/*.tsx`, `components/charts/RangeChart.tsx`
**Description**: ECharts custom `renderItem` functions and tooltip formatters are typed as `(_params: any, api: any)` throughout. This is 29 instances across chart components.

**Evidence**: `grep -rn ": any" --include="*.tsx" --include="*.ts" . | grep -v "node_modules\|\.d\.ts" | wc -l` returns 29 hits, almost all in chart-related code.

ECharts 6's TypeScript types are available but complex. The pattern `(_params: any, api: any)` for custom renderItem is pervasive because ECharts' `CustomSeriesRenderItemParams` and `CustomSeriesRenderItemAPI` types are verbose.

**Knock-on Effects if Addressed**: Would need to import and use ECharts' official render-item types. Risk is type-only -- no runtime behavior change.

**Recommendation**: Create type aliases in `lib/echarts.ts`:
```typescript
export type RenderItemParams = Parameters<NonNullable<CustomSeriesOption['renderItem']>>[0]
export type RenderItemAPI = Parameters<NonNullable<CustomSeriesOption['renderItem']>>[1]
```
Then use them in chart components. Low priority but improves maintainability.

### [QUALITY] `undefined as T` Unsafe Type Assertion in fetchApi

**Severity**: LOW
**Location**: `api/client.ts:711`, `api/client.ts:738`
**Description**: The `fetchApi` function handles HTTP 204 responses by returning `undefined as T`. This is a type-safety hole -- if a caller expects `fetchApi<User[]>` but the server returns 204, they get `undefined` at runtime but TypeScript thinks it is `User[]`.

**Evidence**:
```typescript
// line 711 (retry path)
if (retryResponse.status === 204) return undefined as T
// line 738 (normal path)
if (response.status === 204) return undefined as T
```

**Knock-on Effects if Addressed**: Would need to make the return type `T | undefined` or use a separate `fetchApiVoid` for DELETE operations. This is a type-system change that would propagate to all 26 API namespaces.

**Recommendation**: For now, this is acceptable pragmatism -- all DELETE endpoints are called via mutation hooks where the return value is ignored. Document the contract explicitly. Long-term, consider splitting `fetchApi` and `fetchApiVoid`.
