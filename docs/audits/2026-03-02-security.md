# Security Audit -- Skeptic Review
Date: 2026-03-02
Auditor: Application Security Engineer (automated)
Scope: Full-stack Cassini SPC platform -- backend, frontend, bridge

## Executive Summary

The Cassini codebase demonstrates **above-average security hygiene** for an application at this stage: Argon2id for passwords, bcrypt for API keys, Fernet encryption for stored credentials, HMAC-SHA256 for webhooks with constant-time comparison, DB-backed OIDC state tokens, and structured audit logging. However, the audit identified **22 findings** across the OWASP Top 10 categories, including 2 CRITICAL, 5 HIGH, 9 MEDIUM, and 6 LOW severity issues. The most severe findings are a SQL injection vector in the MySQL OPTIMIZE TABLE path and the absence of JWT revocation, which means compromised tokens remain valid for their full 15-minute lifetime even after password reset or account deactivation.

### [A03: Injection] SQL Injection in MySQL OPTIMIZE TABLE

**Severity**: CRITICAL
**CVSS Estimate**: 8.1
**Location**: `backend/src/cassini/api/v1/database_admin.py:504-505`

**Description**: The `vacuum_database` endpoint for MySQL dialect fetches table names from `information_schema.tables` and interpolates them directly into a SQL `text()` call using an f-string. While the table names come from the database's own metadata (not direct user input), a malicious table name created through any other vector (e.g., a migration, direct DB access, or a compromised admin creating a table with SQL in its name) would allow arbitrary SQL execution.

**Attack Vector**: If an attacker can create a table with a name like `` `; DROP TABLE user; --` `` (via any code path that creates tables, or direct DB access), calling the vacuum endpoint would execute arbitrary SQL.

**Evidence**:
```python
# Line 501-505
result = await session.execute(
    text("SELECT TABLE_NAME FROM information_schema.tables WHERE table_schema = DATABASE()")
)
tables = [row[0] for row in result]
for table in tables:
    await session.execute(text(f"OPTIMIZE TABLE `{table}`"))
```

The backtick escaping is insufficient -- a table name containing a backtick character would break out.

**Knock-on Effects if Addressed**: None meaningful; the fix is purely defensive.

**Recommendation**: Use parameterized queries or validate table names against a strict allowlist pattern:
```python
import re
for table in tables:
    if not re.match(r'^[a-zA-Z_][a-zA-Z0-9_]*$', table):
        logger.warning("skipping_suspicious_table_name", table=table)
        continue
    await session.execute(text(f"OPTIMIZE TABLE `{table}`"))
```

### [A07: Auth Failures] No JWT Revocation -- Tokens Valid After Password Reset/Deactivation

**Severity**: CRITICAL
**CVSS Estimate**: 8.0
**Location**: `backend/src/cassini/core/auth/jwt.py` (entire module), `backend/src/cassini/api/v1/auth.py:533`

**Description**: JWT tokens are stateless with no server-side revocation mechanism. There is no token denylist, no `jti` claim, and no `password_changed_at` check during token verification. When a user's password is reset (line 533: "Note: Refresh tokens are stateless JWTs -- we cannot revoke them from the DB"), changes their password, or has their account deactivated, all previously issued access tokens (15-min) and refresh tokens (7-day) remain valid until natural expiry.

**Attack Vector**:
1. Attacker compromises a user's credentials and obtains JWT tokens
2. User discovers the compromise and resets their password
3. Attacker's tokens remain valid for up to 7 days (refresh) or 15 minutes (access)
4. Attacker can continue accessing the system and refreshing tokens

**Evidence**:
```python
# jwt.py:82-99 -- verify_access_token has no revocation check
def verify_access_token(token: str) -> Optional[dict]:
    try:
        payload = jwt.decode(token, JWT_SECRET_KEY, algorithms=[JWT_ALGORITHM])
        if payload.get("type") != "access":
            return None
        return payload
    except jwt.ExpiredSignatureError:
        return None
    except jwt.InvalidTokenError:
        return None

# auth.py:533 -- explicit acknowledgment of the gap
# Note: Refresh tokens are stateless JWTs -- we cannot revoke them from the DB.
```

**Knock-on Effects if Addressed**: Adds a DB lookup on every authenticated request (or at least on refresh). Consider a Redis-based denylist or embed `password_changed_at` in the JWT and check it during verification.

**Recommendation**:
1. Add a `jti` (JWT ID) claim to refresh tokens and store them in a DB table
2. On password change/reset, delete all refresh tokens for that user
3. Alternatively, embed `password_changed_at` epoch in access tokens and reject tokens issued before the last password change:
```python
def verify_access_token(token: str) -> Optional[dict]:
    payload = jwt.decode(token, JWT_SECRET_KEY, algorithms=[JWT_ALGORITHM])
    if payload.get("type") != "access":
        return None
    # Check iat against user's password_changed_at (requires DB lookup)
    return payload
```

### [A01: Broken Access Control] OIDC Redirect URI Not Validated

**Severity**: HIGH
**CVSS Estimate**: 7.4
**Location**: `backend/src/cassini/api/v1/oidc.py:107-132`, `backend/src/cassini/core/oidc_service.py:64-123`

**Description**: The OIDC `/authorize/{provider_id}` endpoint accepts an arbitrary `redirect_uri` query parameter from the client and passes it directly to the OIDC provider and stores it in the DB state record. There is no validation that the redirect URI belongs to the application's configured domain(s). An attacker could supply a malicious redirect URI pointing to their own server, intercept the authorization code, and use it to obtain tokens.

**Attack Vector**:
1. Attacker crafts a URL: `/api/v1/auth/oidc/authorize/1?redirect_uri=https://evil.com/callback`
2. User clicks the link and authenticates with the OIDC provider
3. Provider redirects to `https://evil.com/callback?code=AUTH_CODE&state=STATE`
4. Attacker intercepts the authorization code
5. If the OIDC provider validates redirect_uri on token exchange, this is mitigated. But many providers are permissive.

**Evidence**:
```python
# oidc.py:107-111 -- redirect_uri taken directly from query params, no validation
@router.get("/authorize/{provider_id}", response_model=OIDCAuthorizationResponse)
async def authorize(
    provider_id: int,
    redirect_uri: str = Query(..., description="Frontend callback URL"),
    session: AsyncSession = Depends(get_db_session),
) -> OIDCAuthorizationResponse:
```

```python
# oidc_service.py:89-93 -- redirect_uri passed directly to OAuth client
oauth_client = AsyncOAuth2Client(
    client_id=config.client_id,
    client_secret=client_secret,
    scope=" ".join(scopes),
    redirect_uri=redirect_uri,  # <-- unvalidated
)
```

**Knock-on Effects if Addressed**: Frontend must send a redirect_uri that matches the configured allowlist. May require a new DB column on OIDCConfig for allowed redirect URIs.

**Recommendation**: Add a `redirect_uris` field to the OIDC config and validate the incoming `redirect_uri` against it:
```python
allowed_uris = config.allowed_redirect_uris  # list from DB
if redirect_uri not in allowed_uris:
    raise HTTPException(status_code=400, detail="Invalid redirect_uri")
```

### [A01: Broken Access Control] WebSocket No Plant-Scoped Authorization

**Severity**: HIGH
**CVSS Estimate**: 7.1
**Location**: `backend/src/cassini/api/v1/websocket.py:290-413`

**Description**: The WebSocket endpoint authenticates the user via JWT but performs no authorization check when subscribing to characteristics. Any authenticated user (even an operator at Plant A) can subscribe to characteristic IDs from Plant B and receive real-time sample data, violations, and acknowledgments for plants they have no access to.

**Attack Vector**:
1. Authenticate as any user (even lowest-privilege operator at one plant)
2. Connect to `/ws/samples?token=JWT`
3. Send `{"type": "subscribe", "characteristic_ids": [1, 2, 3, ..., 1000]}`
4. Receive real-time data from all characteristics across all plants

**Evidence**:
```python
# websocket.py:350-366 -- subscribe handler has no RBAC check
if message_type == "subscribe":
    char_ids = data.get("characteristic_ids")
    # ...
    await manager.subscribe(connection_id, char_ids)  # No plant/role check
```

**Knock-on Effects if Addressed**: Requires resolving plant_id for each characteristic at subscription time and checking user roles. May add latency to subscription operations.

**Recommendation**: Store the authenticated user's ID in the WSConnection and validate plant access on subscribe:
```python
# At connection time, store user_id from the JWT payload
payload = verify_access_token(token)
user_id = int(payload["sub"])

# At subscribe time, check each characteristic's plant against user's roles
for char_id in char_ids:
    plant_id = await resolve_plant_id_for_characteristic(char_id, session)
    check_plant_role(user, plant_id, "operator")
```

### [A01: Broken Access Control] Backup Path Traversal

**Severity**: HIGH
**CVSS Estimate**: 7.0
**Location**: `backend/src/cassini/api/v1/database_admin.py:390-469`

**Description**: The `backup_database` endpoint accepts a `backup_dir` parameter that specifies where to write the backup file. While it checks that the directory exists with `is_dir()`, it does not validate the path against directory traversal. An admin user could write the database backup to any writable directory on the server (e.g., `/tmp`, a web-accessible directory, or overwrite system files via symlinks).

**Attack Vector**:
1. Admin sends `POST /api/v1/database/backup?backup_dir=/var/www/html/public/`
2. The database file (containing all user credentials, secrets, etc.) is copied to a publicly accessible web directory
3. Alternatively: `backup_dir=../../../../etc/cron.d/` could place files in sensitive locations

**Evidence**:
```python
# database_admin.py:419-424
if backup_dir:
    dest_dir = Path(backup_dir)  # <-- arbitrary path from user input
    if not dest_dir.is_dir():
        raise HTTPException(status_code=400, detail=f"Backup directory does not exist: {backup_dir}")
else:
    dest_dir = source.parent
```

**Knock-on Effects if Addressed**: Admins would be restricted to writing backups to a configured backup directory.

**Recommendation**: Remove the `backup_dir` parameter entirely (backups should always go to a configured location), or validate against an allowlist:
```python
ALLOWED_BACKUP_DIR = Path("./backups").resolve()
if backup_dir:
    dest_dir = Path(backup_dir).resolve()
    if not str(dest_dir).startswith(str(ALLOWED_BACKUP_DIR)):
        raise HTTPException(status_code=400, detail="Backup directory outside allowed path")
```

### [A10: SSRF] ERP Connector Base URL Allows Internal Network Scanning

**Severity**: HIGH
**CVSS Estimate**: 6.8
**Location**: `backend/src/cassini/api/v1/erp_connectors.py:184-221`, `backend/src/cassini/core/erp/base.py`, `backend/src/cassini/core/erp/sap_odata.py`, `backend/src/cassini/core/erp/generic_lims.py`

**Description**: The ERP connector system allows admin users to configure arbitrary `base_url` values and then trigger requests to those URLs via `test_connection()` and `fetch_records()`. There is no validation that the URL points to an external host -- an attacker with admin access could use this as an SSRF proxy to scan internal networks, access cloud metadata endpoints (e.g., `http://169.254.169.254/`), or interact with internal services.

**Attack Vector**:
1. Create an ERP connector with `base_url: "http://169.254.169.254/latest/meta-data/iam/security-credentials/"`
2. Call `POST /api/v1/erp/connectors/{id}/test`
3. The server makes the request and may return cloud credentials in error messages

**Evidence**:
```python
# sap_odata.py:25-28 -- direct HTTP request to user-configured URL
async with httpx.AsyncClient(timeout=15.0, verify=True) as client:
    resp = await client.get(f"{self.base_url}/$metadata", headers=headers)
```

```python
# generic_lims.py:30-33 -- same pattern
async with httpx.AsyncClient(timeout=15.0, verify=True) as client:
    resp = await client.get(self.base_url, headers=headers)
```

**Knock-on Effects if Addressed**: May require a URL validation allowlist or a deny-by-default approach for internal IPs.

**Recommendation**: Add URL validation to reject internal/private IP ranges:
```python
import ipaddress
from urllib.parse import urlparse

def validate_external_url(url: str) -> None:
    parsed = urlparse(url)
    hostname = parsed.hostname
    if not hostname:
        raise ValueError("Invalid URL")
    try:
        ip = ipaddress.ip_address(hostname)
        if ip.is_private or ip.is_loopback or ip.is_link_local:
            raise ValueError("Internal addresses not allowed")
    except ValueError:
        pass  # hostname, not IP -- DNS resolution check recommended
```

### [A09: Logging Failures] Error Messages Leak Internal Details via `str(e)`

**Severity**: MEDIUM
**CVSS Estimate**: 5.3
**Location**: Multiple files (see evidence below)

**Description**: The codebase has approximately 15+ instances where `str(e)` from Python exceptions is passed directly to HTTP responses via `HTTPException(detail=str(e))`. This leaks internal implementation details including file paths, SQL error messages, library versions, and stack context to API clients.

**Attack Vector**: An attacker can trigger error conditions and collect information about the internal architecture, database schema, library versions, and file paths to refine further attacks.

**Evidence** (direct `str(e)` to API client):
```
database_admin.py:96   -- raise HTTPException(status_code=400, detail=str(e))
database_admin.py:166  -- raise HTTPException(status_code=400, detail=str(e))
characteristics.py:481 -- detail=str(e)
opcua_servers.py:532   -- detail=str(e)
opcua_servers.py:594   -- detail=str(e)
users.py:211           -- detail=str(e)
erp_connectors.py:738  -- detail=str(e)
erp_connectors.py:829  -- detail=str(e)
brokers.py:306         -- message=f"Connection failed: {str(e)}"
opcua_servers.py:227   -- message=f"Connection failed: {str(e)}"
violations.py:429-438  -- error_msg = str(e) then used in response
samples.py:1148        -- errors.append(f"Sample {idx + 1}: {str(e)}")
```

**Knock-on Effects if Addressed**: Some error messages from `ValueError` are intentionally user-facing (e.g., validation). Need to distinguish between validation errors (safe to surface) and unexpected errors (must not surface).

**Recommendation**: Replace `str(e)` with generic messages for non-validation errors. Keep `ValueError` messages only when they are known-safe:
```python
except ValueError as e:
    raise HTTPException(status_code=400, detail="Validation failed")
except Exception:
    logger.exception("operation_failed")
    raise HTTPException(status_code=500, detail="Internal server error")
```

### [A02: Cryptographic Failures] JWT Secret Auto-Generated and Persisted to Plaintext File

**Severity**: MEDIUM
**CVSS Estimate**: 5.5
**Location**: `backend/src/cassini/core/auth/jwt.py:28-39`

**Description**: When no `CASSINI_JWT_SECRET` environment variable is set, the system auto-generates a random secret and writes it to a plaintext `.jwt_secret` file in the working directory. This file has no explicit permission restrictions and could be readable by other processes or users on the system. In containerized deployments, this file persists across container restarts via volume mounts.

**Attack Vector**:
1. On a shared host or misconfigured container, another process reads `.jwt_secret`
2. Attacker can forge arbitrary JWT tokens for any user

**Evidence**:
```python
if not JWT_SECRET_KEY:
    _secret_file = Path(".jwt_secret")
    if _secret_file.exists():
        JWT_SECRET_KEY = _secret_file.read_text().strip()
    else:
        JWT_SECRET_KEY = secrets.token_urlsafe(64)
        _secret_file.write_text(JWT_SECRET_KEY)  # <-- no chmod, world-readable
```

**Knock-on Effects if Addressed**: Deployments without the env var set would fail to start (which is the correct behavior for production).

**Recommendation**:
1. In production, require `CASSINI_JWT_SECRET` and fail loudly if not set
2. If auto-generation is kept for dev, set file permissions: `_secret_file.chmod(0o600)`
3. Log a warning when using auto-generated secrets

### [A02: Cryptographic Failures] Electronic Signatures Use SHA-256 Without HMAC or Secret

**Severity**: MEDIUM
**CVSS Estimate**: 5.0
**Location**: `backend/src/cassini/core/signature_engine.py:652-684`

**Description**: The electronic signature hashes (`compute_resource_hash`, `compute_signature_hash`) use plain SHA-256 without any server-side secret or HMAC key. Anyone with read access to the database can recompute valid hashes for any resource content. The tamper-detection claim of the signature hash is only valid against external modification of the database -- it provides no protection against a database administrator or attacker with DB access forging signatures.

**Attack Vector**:
1. Attacker gains read/write access to the database
2. Modifies resource content (e.g., FAI report data)
3. Recomputes `resource_hash` and `signature_hash` using the same public algorithm
4. The `verify_signature` endpoint reports the signature as valid

**Evidence**:
```python
def compute_resource_hash(resource_type: str, resource_data: dict) -> str:
    canonical = json.dumps({"type": resource_type, **resource_data}, sort_keys=True, separators=(",", ":"), default=str)
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()  # No secret key

def compute_signature_hash(user_id: int, timestamp: datetime, meaning_code: str, resource_hash: str) -> str:
    canonical = json.dumps({...}, sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(canonical.encode("utf-8")).hexdigest()  # No secret key
```

**Knock-on Effects if Addressed**: Existing signatures would need to be re-computed with the new HMAC-based scheme, requiring a migration.

**Recommendation**: Use HMAC-SHA256 with a server-side secret for signature hashes:
```python
import hmac
def compute_signature_hash(..., secret_key: bytes) -> str:
    canonical = json.dumps({...}, sort_keys=True)
    return hmac.new(secret_key, canonical.encode(), hashlib.sha256).hexdigest()
```

### [A01: Broken Access Control] Verify Email Endpoint Has No Auth -- Account Takeover via Email

**Severity**: MEDIUM
**CVSS Estimate**: 5.5
**Location**: `backend/src/cassini/api/v1/auth.py:624-683`

**Description**: The `verify-email` GET endpoint accepts a token parameter and changes the user's email address. While the token is SHA-256 hashed and has a 24-hour expiry, the endpoint requires no authentication. If an attacker obtains the verification token (e.g., from email logs, a forwarded email, or URL history), they can change a user's email address without being logged in.

**Attack Vector**:
1. User requests email change, verification email sent to new address
2. Attacker intercepts the email or obtains the URL (from browser history, referrer headers, etc.)
3. Attacker visits the verification URL -- no auth check
4. User's email is changed to the attacker-controlled address
5. Attacker can now use "forgot password" to take over the account

**Evidence**:
```python
@router.get("/verify-email")
async def verify_email(
    token: str,
    request: Request,
    session: AsyncSession = Depends(get_db_session),  # No get_current_user dependency
) -> dict:
```

**Knock-on Effects if Addressed**: The verification link in the email would need to redirect to a page where the user logs in first, adding friction.

**Recommendation**: Either require authentication (`Depends(get_current_user)`) or make the token single-use and add rate limiting. At minimum, ensure the token is consumed atomically and verify the `user_id` from the token matches the authenticated user.

### [A09: Logging Failures] Audit Middleware Decodes JWT Without Verification

**Severity**: LOW
**CVSS Estimate**: 3.7
**Location**: `backend/src/cassini/core/audit.py:327-354`

**Description**: The `_extract_user_from_request` function in the audit middleware decodes the JWT payload using base64 without verifying the signature. While this is explicitly documented as "best-effort" and the endpoint's auth dependency handles actual verification, it means audit log entries for failed auth attempts could contain attacker-controlled data (a forged JWT with arbitrary username/user_id).

**Evidence**:
```python
def _extract_user_from_request(request: Request) -> tuple[Optional[int], Optional[str]]:
    """Best-effort JWT user extraction for middleware logging.
    Does NOT validate the token fully -- that's the endpoint's job."""
    # ... base64 decode without signature verification ...
    payload = json.loads(base64.urlsafe_b64decode(payload_b64))
    user_id = int(payload.get("sub", 0)) or None
    username = payload.get("username") or None
```

**Knock-on Effects if Addressed**: Would require either importing and calling `verify_access_token` (adds latency to every request) or accepting the current limitation and documenting it.

**Recommendation**: This is acceptable for audit logging purposes, but add a note in the audit log schema that `username` and `user_id` from middleware-captured entries are unverified for requests that fail auth. Alternatively, mark entries where the response status was 401/403 as having unverified user info.

### [A04: Insecure Design] No Account Lockout After Signature Password Failures

**Severity**: LOW
**CVSS Estimate**: 3.5
**Location**: `backend/src/cassini/core/signature_engine.py:255-260`

**Description**: The electronic signature flow verifies the user's password but does not track failed attempts or lock the account after repeated failures. An attacker who has obtained a valid session could brute-force another user's password through the signature endpoint, which has no rate limiting.

**Evidence**:
```python
# signature_engine.py:255-260
if not verify_password(password, user.hashed_password):
    raise HTTPException(
        status_code=status.HTTP_401_UNAUTHORIZED,
        detail="Invalid password for signature",
    )
# No failed_attempt tracking, no lockout
```

**Knock-on Effects if Addressed**: Would need to reuse the lockout logic from the login flow.

**Recommendation**: Track failed signature attempts and apply the same lockout policy as login:
```python
user.failed_login_count = (user.failed_login_count or 0) + 1
if policy and user.failed_login_count >= policy.max_failed_attempts:
    user.locked_until = now + timedelta(minutes=policy.lockout_duration_minutes)
```
