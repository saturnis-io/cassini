# Backend Audit — Skeptic Review
Date: 2026-03-02
Auditor: Skeptical Senior Developer (backend acquisition review)

## Executive Summary

The Cassini backend is a feature-rich SPC platform that has grown rapidly through sprint-based development. The core SPC engine (`core/engine/`) and statistical foundations (`utils/statistics.py`, `utils/constants.py`) are solid and well-structured. However, the rapid feature accumulation has left behind several categories of technical debt: duplicate abstractions that do the same job (AlertManager vs SPCEngine violation creation), a 200-line `main.py` lifespan function that is effectively a "god function," an unused import layer in `api/v1/__init__.py`, and inconsistent patterns across ERP adapters. The codebase is maintainable but would benefit from a consolidation pass before the next major feature push.

### [ANTI-PATTERN] Duplicate LicenseService Instantiation in `main.py`

**Severity**: MEDIUM
**Location**: `backend/src/cassini/main.py:91-96` and `backend/src/cassini/main.py:443-448`
**Description**: `LicenseService` is instantiated twice in `main.py` — once in the `lifespan()` function (stored on `app.state.license_service`) and again at module level (`_license_svc`) to gate router registration. These are two completely separate instances. If the license file changes between the two instantiation points (unlikely but possible), they could disagree. More importantly, the module-level instance runs at import time, which is before `lifespan()` configures logging, so any warnings from the license loader go to unconfigured logging.

**Evidence**:
```python
# In lifespan() — first instance:
license_service = LicenseService(...)
app.state.license_service = license_service

# At module level — second instance:
_license_svc = LicenseService(...)
if _license_svc.is_commercial:
    app.include_router(anomaly_router)
    # ...
```

**Knock-on Effects if Addressed**: Router registration must happen before the app starts, but after settings are loaded. Moving to a factory pattern or pre-computing the boolean at settings load time would require restructuring the initialization order. Alternatively, the lifespan instance could be eliminated and only the module-level one used.
**Recommendation**: Use a single `LicenseService` instance. The simplest fix: use `_license_svc` everywhere and store it on `app.state` in `lifespan()` instead of creating a new one. Alternatively, compute `is_commercial` once from settings and pass the boolean.

### [ANTI-PATTERN] God Function: `lifespan()` at 300+ Lines

**Severity**: MEDIUM
**Location**: `backend/src/cassini/main.py:81-387`
**Description**: The `lifespan()` async context manager is ~300 lines long and does everything: database initialization, admin bootstrap, migration checking, WebSocket setup, MQTT initialization, OPC-UA initialization, anomaly detector setup, forecasting engine setup, notification dispatcher setup, push service setup, ERP sync setup, audit service wiring (including 6 inline event handler closures), purge engine setup, and report scheduler setup. This makes it extremely difficult to test any single initialization step in isolation, and a failure in one section can cascade unpredictably.

**Evidence**: The function defines 6 inline async closure functions (`_audit_violation_created`, `_audit_limits_updated`, `_audit_char_created`, `_audit_char_deleted`, `_audit_anomaly_detected`, `_audit_erp_sync_completed`) that subscribe to the event bus. These closures capture `audit_service` from the enclosing scope and cannot be tested independently.

**Knock-on Effects if Addressed**: Extracting initialization into separate functions (e.g., `_init_mqtt()`, `_init_commercial_services()`, `_init_audit_subscriptions()`) would be a low-risk refactor. The inline closures could move to methods on `AuditService` itself.
**Recommendation**: Extract each initialization block into a named async function. Move audit event subscriptions into `AuditService.setup_subscriptions(event_bus)`.

### [ANTI-PATTERN] Duplicate Violation Creation: AlertManager vs SPCEngine

**Severity**: MEDIUM
**Location**: `backend/src/cassini/core/alerts/manager.py:176-251` and `backend/src/cassini/core/engine/spc_engine.py:656-728`
**Description**: Both `AlertManager.create_violations()` and `SPCEngine._create_violations()` create violation records from rule results. They have overlapping but different implementations:
- `AlertManager` creates `Violation` objects directly via `session.add()`, bypasses the `ViolationRepository`, and publishes `ViolationCreated` events (a local dataclass).
- `SPCEngine` uses `ViolationRepository.create()` and publishes `ViolationCreatedEvent` (the event bus event).

The `AlertManager` is injected via the `get_alert_manager` dependency and used by the violations router for acknowledgment. However, the SPCEngine never uses `AlertManager` for violation creation — it has its own path. This means there are two parallel violation creation codepaths that could diverge in behavior (and have: `SPCEngine` supports `requires_acknowledgement` per-rule, `AlertManager` does not).

**Evidence**:
```python
# AlertManager.create_violations() — bypasses repo, uses local event type
violation = Violation(sample_id=sample_id, ...)
self._violation_repo.session.add(violation)

# SPCEngine._create_violations() — uses repo, uses event bus event
violation_record = await self._violation_repo.create(
    sample_id=sample_id, char_id=characteristic_id, ...
    requires_acknowledgement=requires_ack,
)
await self._event_bus.publish(ViolationCreatedEvent(...))
```

**Knock-on Effects if Addressed**: `AlertManager.create_violations()` is now only used if something still calls it outside the SPC engine. Given that `SPCEngine._create_violations` is the actual hot path, `AlertManager.create_violations` may be dead code. The `AlertManager` is still used for acknowledgment and stats — those should remain.
**Recommendation**: Audit whether `AlertManager.create_violations()` is called anywhere. If not, remove it. If it is, consolidate both paths into a single `ViolationService` that handles creation consistently.

### [ANTI-PATTERN] SampleContext Receives `metadata` But Dataclass Has No Such Field

**Severity**: HIGH
**Location**: `backend/src/cassini/api/v1/data_entry.py:208` and `backend/src/cassini/core/providers/protocol.py:14-27`
**Description**: In `data_entry.py`, the `SampleContext` is constructed with `metadata=data.metadata`, but the `SampleContext` dataclass (in `core/providers/protocol.py`) has no `metadata` field. Python dataclasses silently reject unexpected keyword arguments by raising `TypeError` at runtime. This means the `/data-entry/submit` endpoint and `/data-entry/batch` endpoint will **crash at runtime** whenever the request includes a non-null `metadata` field.

**Evidence**:
```python
# data_entry.py:203-208
context = SampleContext(
    batch_number=data.batch_number,
    operator_id=data.operator_id,
    source="API",
    metadata=data.metadata,  # <-- SampleContext has no 'metadata' field
)

# protocol.py — SampleContext definition:
@dataclass
class SampleContext:
    batch_number: str | None = None
    operator_id: str | None = None
    product_code: str | None = None
    source: str = "MANUAL"
    # NO metadata field
```

**Knock-on Effects if Addressed**: Adding a `metadata: dict | None = None` field to `SampleContext` is safe; it's a dataclass with keyword defaults. Alternatively, remove the `metadata=` kwarg from the two call sites in `data_entry.py`.
**Recommendation**: Either add `metadata: dict | None = None` to `SampleContext` and propagate it to the Sample model, or remove the `metadata` parameter from the two `SampleContext(...)` calls in `data_entry.py`. Given that the `DataEntryRequest` schema accepts metadata and the Sample model does not store it, removing is the safer option until metadata storage is intentionally designed.

### [ANTI-PATTERN] ERP Adapters Duplicate OAuth2 Client Credentials Logic

**Severity**: LOW

**Recommendation**: Extract common OAuth2 flows into `BaseERPAdapter` protected methods. Each adapter's `authenticate()` becomes a dispatcher that calls the appropriate base method.

### [QUESTIONABLE DESIGN] N+1 Query in `violations.py::build_hierarchy_path()`

while current_id is not None:

### [QUESTIONABLE DESIGN] Dual ZoneBoundaries Dataclasses

**Severity**: LOW
**Location**: `backend/src/cassini/utils/statistics.py:46-69` and `backend/src/cassini/core/engine/rolling_window.py:79-96`
**Description**: There are two `ZoneBoundaries` dataclasses: one in `utils/statistics.py` (7 fields) and one in `core/engine/rolling_window.py` (8 fields, extends the first by adding `sigma`). The `spc_engine.py` creates the rolling_window version by first calling `calculate_zones()` (which returns the utils version) then manually copying all 7 fields plus adding `sigma`. This is fragile — if a field is added to the base, the manual copy in `spc_engine.py` breaks silently.

**Evidence**:
```python
# spc_engine.py:626-634 — manual field-by-field copy:
zones = calculate_zones(center_line, sigma)
return ZoneBoundaries(
    center_line=zones.center_line,
    plus_1_sigma=zones.plus_1_sigma,
    plus_2_sigma=zones.plus_2_sigma,
    plus_3_sigma=zones.plus_3_sigma,
    minus_1_sigma=zones.minus_1_sigma,
    minus_2_sigma=zones.minus_2_sigma,
    minus_3_sigma=zones.minus_3_sigma,
    sigma=sigma,
)
```

**Knock-on Effects if Addressed**: Since the rolling_window `ZoneBoundaries` inherits from the base, `calculate_zones()` could be changed to accept an optional `sigma` parameter and return the extended version directly. This eliminates the manual copy.
**Recommendation**: Either (a) add `sigma` to the base `ZoneBoundaries` in `utils/statistics.py` and remove the subclass, or (b) have `calculate_zones()` return the rolling_window version directly. Option (a) is simpler.

### [ANTI-PATTERN] `data_entry.py` Creates SPCEngine Per Request

**Severity**: MEDIUM
**Location**: `backend/src/cassini/api/v1/data_entry.py:52-73`
**Description**: The `get_spc_engine()` helper in `data_entry.py` creates a new `SPCEngine` instance (along with `NelsonRuleLibrary`, `RollingWindowManager`, and three repositories) for every single request to `/data-entry/submit` and `/data-entry/batch`. The `RollingWindowManager` maintains an in-memory LRU cache of rolling windows; creating a new one per request means the cache is always cold and the database is hit for window loading on every single sample submission. For high-frequency data entry (e.g., automated gage readings), this is a significant performance problem.

**Evidence**:
```python
async def get_spc_engine(session: AsyncSession) -> SPCEngine:
    """Create SPC engine instance with all dependencies."""
    sample_repo = SampleRepository(session)
    char_repo = CharacteristicRepository(session)
    violation_repo = ViolationRepository(session)
    window_manager = RollingWindowManager(sample_repo)  # Cold cache every time
    rule_library = NelsonRuleLibrary()
    return SPCEngine(...)
```

Compare with `TagProviderManager.initialize()` which creates a single `SPCEngine` and `RollingWindowManager` at startup and reuses them.

**Knock-on Effects if Addressed**: The `RollingWindowManager` would need to be a shared singleton (it already is in `TagProviderManager`). The session dependency makes this tricky — repositories are session-scoped but the window manager is long-lived. A per-request repo injection with a shared window manager is the correct pattern.
**Recommendation**: Make `RollingWindowManager` a shared instance (similar to `TagProviderManager`). Pass session-scoped repos into the engine but reuse the window manager across requests.

### [QUESTIONABLE DESIGN] Health Check Returns 200 for Unhealthy State

**Severity**: LOW
**Location**: `backend/src/cassini/main.py:486-495`
**Description**: The root `/health` endpoint returns HTTP 200 even when the database is unreachable, just with `{"status": "unhealthy"}`. Load balancers and container orchestrators (Docker, Kubernetes) typically use the HTTP status code to determine health. Returning 200 for an unhealthy state means the load balancer will keep routing traffic to a broken instance.

**Evidence**:
```python
@app.get("/health")
async def health_check() -> dict[str, str]:
    try:
        db = get_database()
        async with db.session() as session:
            await session.execute(text("SELECT 1"))
        return {"status": "healthy"}
    except Exception:
        return {"status": "unhealthy"}  # Still returns 200!
```

**Knock-on Effects if Addressed**: Container healthchecks that rely on status code (the default for Docker HEALTHCHECK and Kubernetes livenessProbe) would start correctly detecting unhealthy instances. Any monitoring that currently keys on the response body would need updating.
**Recommendation**: Return HTTP 503 (Service Unavailable) when the database check fails. Use `JSONResponse(status_code=503, content={"status": "unhealthy"})`.

### [CODE QUALITY] `Notification.send_test_webhook` Does Not Decrypt Secret

**Severity**: MEDIUM
**Location**: `backend/src/cassini/core/notifications.py:574-602`
**Description**: The `send_test_webhook()` static method signs the test payload using the raw `secret` parameter directly, without decrypting it first. However, webhook secrets are stored encrypted in the database (using Fernet). The production `_send_webhook()` method (line 483) correctly decrypts the secret before HMAC signing. This means test webhook payloads will have incorrect HMAC signatures if the secret was stored encrypted, causing the receiving system to reject the test but accept real notifications (or vice versa).

**Evidence**:
```python
# Production path — correctly decrypts:
if secret:
    try:
        key = get_encryption_key()
        decrypted_secret = decrypt_password(secret, key)
    except Exception:
        decrypted_secret = secret
    sig = hmac.new(decrypted_secret.encode(), ...)

# Test path — uses raw (possibly encrypted) secret:
if secret:
    sig = hmac.new(secret.encode("utf-8"), ...)  # NO decryption!
```

**Knock-on Effects if Addressed**: The test endpoint caller would need to either pass a decrypted secret or the `send_test_webhook` method would need to handle decryption. Since this is a static method and callers may pass either form, the safest approach is to add try/except decryption matching the production path.
**Recommendation**: Add the same `decrypt_password` try/except block to `send_test_webhook()`.

### [QUESTIONABLE DESIGN] `Event` Base Class is ABC but Not a Dataclass

**Severity**: LOW
**Location**: `backend/src/cassini/core/events/events.py:13-24`
**Description**: The `Event` base class is defined as an `ABC` with a `timestamp: datetime` attribute annotation, but it is not decorated with `@dataclass`. All subclasses are `@dataclass` classes. This means the `timestamp` annotation on `Event` is just a class-level type hint, not a field — it has no default value and is not included in `__init__`. Each subclass must independently declare `timestamp: datetime = field(...)`, which they all do. The `ABC` base class provides no actual enforcement or shared behavior.

**Evidence**:
```python
class Event(ABC):
    """Base class for all events."""
    timestamp: datetime  # Just an annotation, not a field

@dataclass
class SampleProcessedEvent(Event):
    # Must re-declare timestamp:
    timestamp: datetime = field(default_factory=lambda: datetime.now(timezone.utc))
```

**Knock-on Effects if Addressed**: Making `Event` a `@dataclass` with a default `timestamp` field would eliminate the redundant `timestamp` declaration in every subclass. However, dataclass inheritance with default fields in the parent requires all child fields with defaults to come after those without defaults, which the current subclasses already handle.
**Recommendation**: Consider making `Event` a `@dataclass` with `timestamp` as a field with a default factory. This would reduce 15+ lines of boilerplate across all event classes.

### [ANTI-PATTERN] Audit Middleware Reads Entire Request Body Into Memory

**Severity**: MEDIUM
**Location**: `backend/src/cassini/core/audit.py:252-262`
**Description**: The `AuditMiddleware` reads the entire request body via `await request.body()` for all JSON mutating requests (POST/PUT/PATCH). For most API calls this is fine, but for endpoints that accept large payloads (e.g., `/import/upload` which accepts CSV/Excel files, or `/data-entry/batch` which can have 1000 samples), this doubles the memory consumption — the body is read once by the middleware and then again by FastAPI's request parsing.

**Evidence**:
```python
if request.method in ("POST", "PUT", "PATCH"):
    content_type = request.headers.get("content-type", "")
    if "application/json" in content_type:
        try:
            raw = await request.body()  # Reads entire body
            if raw:
                cached_body = _json.loads(raw)
```

Note: `request.body()` in Starlette caches the body after the first read, so it's not a double-read problem — it's a memory problem for large payloads being stored as a parsed dict in the middleware scope.

**Knock-on Effects if Addressed**: Adding a body size check (e.g., skip body capture for Content-Length > 100KB) would prevent memory spikes. The audit log detail would lose the body for large requests, which is acceptable since the important info is the URL, method, and user.
**Recommendation**: Add a size guard: skip body parsing if `Content-Length` exceeds a threshold (e.g., 100KB). The import endpoint already uses `multipart/form-data` so it's already excluded by the `application/json` check, but `/data-entry/batch` is JSON.

### [QUESTIONABLE DESIGN] TagProviderManager Stores Session After Initialization

**Severity**: MEDIUM
**Location**: `backend/src/cassini/core/providers/manager.py:72,120`
**Description**: `TagProviderManager` stores the `AsyncSession` from initialization (`self._session = session`) and creates repositories bound to that session. FastAPI async sessions are meant to be short-lived (per-request). Storing a session from the `lifespan()` startup means the TagProvider's repositories are using a long-lived session that may become stale, have expired transactions, or accumulate ORM identity map bloat over time. The SPC engine created in `initialize()` will use this same stale session for all subsequent sample processing.

**Evidence**:
```python
async def initialize(self, session: AsyncSession) -> bool:
    self._session = session  # Stored indefinitely
    # Creates repos bound to this session:
    char_repo = CharacteristicRepository(session)
    sample_repo = SampleRepository(session)
    # ...
    self._spc_engine = SPCEngine(char_repo=char_repo, ...)
```

**Knock-on Effects if Addressed**: Requires restructuring the provider manager to accept a session factory and create per-operation sessions. This is a larger refactor but would prevent connection pool exhaustion and stale data issues in long-running deployments.
**Recommendation**: Accept a `session_factory` (like `NotificationDispatcher` and `MQTTPublisher` do) instead of a single session. Create a fresh session for each sample processing operation.
